# State & Priority Scheduling

RyunixJS introduces a **smart priority-based scheduling system** on top of its state dispatcher, ensuring that critical user interactions don't drop frames while heavy renders execute.

## Priority Levels

The `Priority` enum defines five tiers of urgency.

**From `priority.js:4`:**

```javascript
const Priority = {
  IMMEDIATE: 1,       // User input (clicks, typing)
  USER_BLOCKING: 2,   // Hover events, fast scrolls
  NORMAL: 3,          // Data fetching, generic updates
  LOW: 4,             // Analytics events, prefetching
  IDLE: 5,            // Off-screen background tasks
}
```

<Info>
Lower numbers = higher priority. `IMMEDIATE` updates run first, `IDLE` updates run when the browser is completely idle.
</Info>

### Priority Examples

| Priority Level | Use Cases | Examples |
|---------------|-----------|----------|
| **IMMEDIATE** | User input | Typing, clicking buttons, dragging |
| **USER_BLOCKING** | Hover/scroll | Tooltip display, scroll listeners |
| **NORMAL** | Data updates | API responses, form submissions |
| **LOW** | Non-urgent work | Analytics, prefetching routes |
| **IDLE** | Background tasks | Cache cleanup, offline sync |

## Priority Scheduling

### scheduleUpdate()

Pushes a state update with its urgency into the pending queue.

**From `priority.js:21`:**

```javascript
const scheduleUpdate = (callback, priority = Priority.NORMAL) => {
  pendingUpdates.push({ callback, priority, timestamp: Date.now() })

  if (!isScheduling) {
    isScheduling = true
    rIC(processPendingUpdates)  // requestIdleCallback
  }
}
```

**Features:**
- Queues updates with priority metadata
- Uses `requestIdleCallback` for efficient scheduling
- Batches multiple updates for single flush

### processPendingUpdates()

Sorts the update queue by priority and executes highest-priority tasks first.

**From `priority.js:33`:**

```javascript
const processPendingUpdates = (deadline) => {
  pendingUpdates.sort((a, b) => a.priority - b.priority)

  while (pendingUpdates.length > 0 && deadline.timeRemaining() > 1) {
    const update = pendingUpdates.shift()
    currentPriority = update.priority
    update.callback()
  }

  if (pendingUpdates.length > 0) {
    rIC(processPendingUpdates)  // Continue processing
  } else {
    isScheduling = false
    currentPriority = Priority.NORMAL
  }
}
```

<Accordion title="How does priority sorting work?">
The queue is sorted by `priority` value (ascending):

```javascript
pendingUpdates.sort((a, b) => a.priority - b.priority)
```

This ensures:
- `IMMEDIATE (1)` tasks run before `NORMAL (3)`
- `NORMAL (3)` tasks run before `LOW (4)`
- `LOW (4)` tasks run before `IDLE (5)`

Within same priority, tasks execute in insertion order (FIFO).
</Accordion>

### runWithPriority()

Temporarily elevates context priority while executing a callback.

**From `priority.js:53`:**

```javascript
const runWithPriority = (priority, callback) => {
  const previousPriority = currentPriority
  currentPriority = priority

  try {
    return callback()
  } finally {
    currentPriority = previousPriority
  }
}
```

**Usage:**

```javascript
import { runWithPriority, Priority } from '@unsetsoft/ryunixjs'

runWithPriority(Priority.IMMEDIATE, () => {
  // This code runs with IMMEDIATE priority
  updateCriticalUI()
})
```

## Hook Implementations

### useStorePriority()

A low-level state dispatcher with priority support.

**From `hooks.js:653`:**

```javascript
const useStorePriority = (initialState) => {
  const reducer = (state, action) =>
    typeof action === 'function' ? action.value(state) : action.value

  const [state, baseDispatch] = useReducer(reducer, initialState)

  const dispatch = (action, priority = Priority.NORMAL) => {
    const wrappedAction = {
      value: action,
      priority,
    }

    scheduleUpdate(() => baseDispatch(wrappedAction), priority)
  }

  return [state, dispatch]
}
```

**Usage:**

```javascript
import { useStorePriority, Priority } from '@unsetsoft/ryunixjs'

function SearchInput() {
  const [query, setQuery] = useStorePriority('')

  const handleChange = (e) => {
    // High priority: Keep input responsive
    setQuery(e.target.value, Priority.IMMEDIATE)
  }

  return <input value={query} onChange={handleChange} />
}
```

### useTransition()

Marks updates as non-urgent to keep the UI responsive during heavy renders.

**From `hooks.js:674`:**

```javascript
const useTransition = () => {
  const [isPending, setIsPending] = useStorePriority(false)

  const startTransition = (callback) => {
    setIsPending(true, Priority.IMMEDIATE)

    setTimeout(() => {
      runWithPriority(Priority.LOW, () => {
        callback()
        setIsPending(false, Priority.LOW)
      })
    }, 0)
  }

  return [isPending, startTransition]
}
```

**Real-world example:**

```javascript
import { useTransition, useStore } from '@unsetsoft/ryunixjs'

function TabContainer() {
  const [tab, setTab] = useStore('home')
  const [isPending, startTransition] = useTransition()

  const switchTab = (newTab) => {
    startTransition(() => {
      setTab(newTab)  // Heavy render, marked as LOW priority
    })
  }

  return (
    <div>
      <button onClick={() => switchTab('home')}>Home</button>
      <button onClick={() => switchTab('profile')}>Profile</button>
      <button onClick={() => switchTab('settings')}>Settings</button>
      
      {isPending && <Spinner />}
      
      {tab === 'home' && <HeavyHomePage />}
      {tab === 'profile' && <HeavyProfilePage />}
      {tab === 'settings' && <HeavySettingsPage />}
    </div>
  )
}
```

<Info>
`useTransition` dispatches the pending state as `IMMEDIATE` (for instant spinner) but schedules the actual work as `LOW` priority, yielding the main thread for user interactions.
</Info>

### useDeferredValue()

Defers updates with a 100ms debounce and `Priority.LOW`.

**From `hooks.js:694`:**

```javascript
const useDeferredValue = (value) => {
  const [deferredValue, setDeferredValue] = useStorePriority(value)

  useEffect(() => {
    const timeout = setTimeout(() => {
      setDeferredValue(value, Priority.LOW)
    }, 100)

    return () => clearTimeout(timeout)
  }, [value])

  return deferredValue
}
```

**Usage:**

```javascript
import { useDeferredValue, useStore } from '@unsetsoft/ryunixjs'

function SearchPage() {
  const [searchTerm, setSearchTerm] = useStore('')
  const deferredSearchTerm = useDeferredValue(searchTerm)

  // searchTerm updates immediately (input stays responsive)
  // deferredSearchTerm updates after 100ms with LOW priority

  return (
    <div>
      <input 
        value={searchTerm} 
        onChange={e => setSearchTerm(e.target.value)} 
        placeholder="Search..."
      />
      <ExpensiveSearchResults query={deferredSearchTerm} />
    </div>
  )
}
```

<Tip>
Use `useDeferredValue` for expensive computations that depend on fast-changing values like search inputs or sliders.
</Tip>

## Update Batching

### batchUpdates()

Groups multiple synchronous state updates into a single render cycle.

**From `batching.js:34`:**

```javascript
const batchUpdates = (callback) => {
  if (isBatching) {
    callback()  // Already batching, just execute
    return
  }

  isBatching = true
  try {
    callback()
  } finally {
    isBatching = false
    flushUpdates()  // Trigger single render
  }
}
```

**How it works:**

1. When `isBatching = true`, `queueUpdate()` stalls updates in `pendingUpdates` array
2. Multiple state updates accumulate without triggering renders
3. `finally` block calls `flushUpdates()` which triggers a single `scheduleWork()`

**Example:**

```javascript
import { batchUpdates } from '@unsetsoft/ryunixjs'

function MultiUpdate() {
  const [count, setCount] = useStore(0)
  const [text, setText] = useStore('')
  const [enabled, setEnabled] = useStore(false)

  const handleClick = () => {
    batchUpdates(() => {
      setCount(c => c + 1)      // Queued
      setText('Updated')         // Queued
      setEnabled(true)           // Queued
      // Single render after all updates
    })
  }

  return <button onClick={handleClick}>Update All</button>
}
```

<Info>
RyunixJS automatically batches updates inside event handlers. Use `batchUpdates()` explicitly for async contexts like `setTimeout` or promise callbacks.
</Info>

## Work Loop Priority Integration

The work loop respects priority when scheduling work.

**From `workers.js:129`:**

```javascript
const scheduleWork = (root, priority = Priority.NORMAL) => {
  const state = getState()

  if (state.wipRoot) {
    workQueue.push(root)
  } else {
    state.nextUnitOfWork = root
    state.wipRoot = root
    state.deletions = []
  }

  state.hookIndex = 0
  state.effects = []

  if (!isWorkLoopScheduled) {
    isWorkLoopScheduled = true
    if (priority <= Priority.USER_BLOCKING) {
      rIC(workLoop)  // Immediate scheduling
    } else {
      setTimeout(() => rIC(workLoop), 0)  // Defer low priority
    }
  }
}
```

<Accordion title="How does priority affect scheduling?">
- **High priority** (`IMMEDIATE`, `USER_BLOCKING`): Calls `rIC(workLoop)` immediately
- **Low priority** (`NORMAL`, `LOW`, `IDLE`): Defers via `setTimeout(() => rIC(workLoop), 0)`

This ensures urgent updates start processing ASAP while low-priority updates wait for idle time.
</Accordion>

## Performance Patterns

### Pattern 1: Responsive Input

```javascript
function FormField() {
  const [value, setValue] = useStore('')
  const [suggestions, setSuggestions] = useStore([])
  const [isPending, startTransition] = useTransition()

  const handleChange = (e) => {
    setValue(e.target.value)  // Immediate: Keep input responsive

    startTransition(() => {
      // Low priority: Heavy suggestion filtering
      const filtered = hugeSuggestionList.filter(s => 
        s.toLowerCase().includes(e.target.value.toLowerCase())
      )
      setSuggestions(filtered)
    })
  }

  return (
    <div>
      <input value={value} onChange={handleChange} />
      {isPending && <Spinner />}
      <SuggestionList items={suggestions} />
    </div>
  )
}
```

### Pattern 2: Tab Switching

```javascript
function TabPanel() {
  const [activeTab, setActiveTab] = useStore(0)
  const [isPending, startTransition] = useTransition()

  const handleTabClick = (index) => {
    startTransition(() => {
      setActiveTab(index)  // Heavy re-render marked as LOW
    })
  }

  return (
    <div>
      <Tabs onClick={handleTabClick} />
      {isPending ? <Skeleton /> : <TabContent tab={activeTab} />}
    </div>
  )
}
```

### Pattern 3: Deferred Search

```javascript
function DataGrid({ data }) {
  const [filter, setFilter] = useStore('')
  const deferredFilter = useDeferredValue(filter)

  const filteredData = useMemo(() => {
    return data.filter(row => 
      Object.values(row).some(val => 
        String(val).includes(deferredFilter)
      )
    )
  }, [data, deferredFilter])  // Only recomputes after 100ms debounce

  return (
    <div>
      <input 
        value={filter} 
        onChange={e => setFilter(e.target.value)}
        placeholder="Filter data..."
      />
      <Table data={filteredData} />
    </div>
  )
}
```

### Pattern 4: Analytics (IDLE Priority)

```javascript
function TrackingWrapper({ children }) {
  const [events, setEvents] = useStorePriority([])

  const trackEvent = (eventName, data) => {
    setEvents(prev => [...prev, { eventName, data, timestamp: Date.now() }], Priority.IDLE)
  }

  useEffect(() => {
    if (events.length >= 10) {
      // Batch send analytics when idle
      runWithPriority(Priority.IDLE, () => {
        sendAnalytics(events)
        setEvents([], Priority.IDLE)
      })
    }
  }, [events])

  return <EventContext.Provider value={trackEvent}>{children}</EventContext.Provider>
}
```

## Best Practices

<Card title="User Input = IMMEDIATE" icon="keyboard">
  Always use `IMMEDIATE` or `USER_BLOCKING` priority for direct user interactions (typing, clicking, dragging).
</Card>

<Card title="Heavy Renders = LOW" icon="weight">
  Wrap expensive state updates in `startTransition()` to prevent UI blocking.
</Card>

<Card title="Analytics = IDLE" icon="chart-line">
  Use `IDLE` priority for non-critical background tasks like analytics, logging, and cache cleanup.
</Card>

<Card title="Batch Updates" icon="layer-group">
  Use `batchUpdates()` in async contexts to prevent multiple re-renders.
</Card>
